## 1. SYSTEM IDENTITY

V2H-Rectify is a full-stack OCR system for Traditional Chinese handwritten
manuscript paper.

Main capabilities:
1. Corner detection and perspective correction
2. OCR pipeline with CRAFT + OpenCV preprocessing + VLM recognition
3. Real-time progress over WebSocket
4. Single-image and batch OCR workflows
5. Frontend UI for corner editing, progress monitoring, and result export

-------------------------------------------------------------------------------

## 2. RUNTIME ARCHITECTURE

### 2.1 Service topology

- frontend: Nginx static host and reverse proxy
  - host port: 3000
- backend: FastAPI app
  - host port: 8080
- vllm: OpenAI-compatible inference backend
  - host port: 8001

### 2.2 Network/proxy assumptions

- Frontend dev server proxies /api to http://localhost:8080
- WebSocket endpoints are also under /api/v1/ws/*
- Vite polling watch is enabled to avoid container inotify limits
  - usePolling: true
  - interval: 300ms

-------------------------------------------------------------------------------

## 3. HIGH-LEVEL FLOWS

### 3.1 Single OCR flow

1. POST /api/v1/corner/detect
2. Optional manual correction in UI
3. POST /api/v1/corner/correct
4. POST /api/v1/ocr/upload
5. WS /api/v1/ws/{task_id} for stage/progress
6. GET /api/v1/ocr/{task_id} for fallback/final result

### 3.2 Batch OCR flow

1. POST /api/v1/ocr/batch/prepare
   - accepts up to 50 files
   - returns per-image task_id + auto corners + confidence
2. Optional per-image corner edits in frontend state
3. POST /api/v1/ocr/batch/submit with final corners
4. WS /api/v1/ws/batch/{batch_id} for aggregated batch progress
5. GET /api/v1/ocr/batch/{batch_id} for full per-task results

## 5. BACKEND MODULE DETAILS

### 5.1 api/app.py

- FastAPI title: Traditional Chinese Handwriting OCR
- FastAPI version: 3.0.0
- CORS: allow_origins=["*"], allow_credentials=True, allow_methods=["*"], allow_headers=["*"]
- Routers mounted under /api/v1:
  - health
  - corner
  - ocr
  - ws

Lifespan startup:
1. load settings
2. create CRAFTService (device from settings.cuda_device)
3. start CRAFTService worker
4. create VLMService.from_settings
5. create OCRPipeline
6. register services in deps

Lifespan shutdown:
1. stop CRAFTService
2. clear deps singletons

### 5.2 api/deps.py pattern

- Module-level singleton storage
- get_craft(), get_vlm(), get_pipeline() used with FastAPI Depends

### 5.3 api/schemas.py (contract source of truth)

Enums:
- TaskStatus: pending | processing | completed | failed

Single OCR models:
- Point(x, y)
- CornerDetectResponse(task_id, corners[4], confidence, preview_url)
- CornerCorrectRequest(task_id, corners[4])
- CornerCorrectResponse(corrected, preview_url)
- OCROptions(auto_rotate, remove_print, auto_split)
- OCRSubmitResponse(task_id, status)
- ColumnData(col_index, text, spacing_indexes, num_rows)
- OCRStatusResponse(task_id, status, progress, title, text, columns,
  rotation_angle, elapsed_seconds, error)

Batch models:
- BatchTaskItem(task_id, filename, status)
- BatchSubmitResponse(batch_id, tasks[])
- BatchStatusResponse(batch_id, total, completed, failed, processing,
  progress, tasks[])
- BatchPrepareItem(task_id, filename, corners, confidence)
- BatchPrepareResponse(items[])
- BatchCorrectionItem(task_id, corners)
- BatchSubmitRequest(corrections[], auto_rotate, remove_print, auto_split)

Health:
- HealthResponse(status, craft_loaded, vlm_backend, version)

WebSocket payloads:
- WSProgressMessage(task_id, stage, progress, message, status)
- WSBatchProgressMessage(batch_id, progress, completed, failed, total, status)

### 5.4 api/ws.py

Connection registry:
- _connections: Dict[str, Set[WebSocket]]
- Single key: task_id
- Batch key: "batch:{batch_id}"

Endpoints:
- /ws/{task_id}
- /ws/batch/{batch_id}

Broadcast functions:
- broadcast_progress(...)
- broadcast_batch_progress(...)

Behavior:
- sends JSON payload from Pydantic model_dump_json
- drops dead sockets silently
- keeps socket alive by waiting for receive_text in loop

### 5.5 api/routes/corner.py

In-memory store:
- _pending_images: task_id -> BGR image

POST /corner/detect:
- decode upload with cv2.imdecode
- runs detect_corners
- fallback to full-image corners if:
  - detection failed OR
  - contour area ratio < 0.35
- confidence is 1.0 when valid, else 0.0
- stores original image in _pending_images[task_id]

POST /corner/correct:
- pops image by task_id
- warps with provided corners
- stores warped image back into _pending_images[task_id]

### 5.6 api/routes/ocr.py

In-memory stores:
- _tasks: task_id -> status/progress/result/error
- _batches: batch_id -> {task_ids, filenames}

Single route behavior:
- POST /ocr/upload
  - if task_id provided and exists in _pending_images, consumes warped image
  - else decodes uploaded file as new task
  - sets _tasks[task_id] to processing
  - executes _run_ocr in executor thread

- GET /ocr/{task_id}
  - returns task status/result
  - 404 if task not found

Batch route behavior:
- POST /ocr/batch/prepare
  - files required, max 50
  - skips invalid images
  - corner detect per image with same 0.35 fallback rule
  - stores originals in _pending_images
  - returns items[] with task_id/filename/corners/confidence

- POST /ocr/batch/submit
  - requires corrections[]
  - for each correction:
    - pop original from _pending_images
    - warp using provided corners
    - set _tasks[task_id] processing
    - run _run_batch_ocr in executor
  - creates _batches[batch_id] with task list
  - note: returned BatchTaskItem.filename is currently empty string

- GET /ocr/batch/{batch_id}
  - aggregates completed/failed/processing/progress
  - returns per-task OCRStatusResponse list

Thread/async bridge details:
- Executor thread updates _tasks
- Progress bridge uses asyncio.run_coroutine_threadsafe to call async WS broadcast

### 5.7 services/ocr_pipeline.py

Public API:
- run(image, auto_rotate?, remove_print?, auto_split?, debug?, debug_tag?, on_progress?)

Pipeline design highlights:
- single CRAFT pass + heatmap reuse where possible
- optional image splitting before processing
- optional rotation correction
- optional print removal
- grid extraction
- vertical-to-horizontal reformat
- concurrent VLM OCR per column
- final text merge and title extraction

Single-part progress mapping (core):
- 0.03, 0.06: image analysis/splitting
- 0.05-0.20: CRAFT detect
- 0.22-0.35: rotation
- 0.37-0.45: print removal
- 0.47-0.55: grid extraction
- 0.57-0.62: text formatting
- 0.63-0.93: VLM column OCR
- 0.97: text merge

Returned OCRResult fields:
- text
- title
- columns[]
- rotation_angle
- print_removed
- was_split
- num_columns
- elapsed_seconds
- debug_dir

### 5.8 core/text_reformat.py

Key constants:
- _LINE_THICKNESS_RATIO = 0.10
- _LINE_SPAN_RATIO = 0.50
- _HEAT_ACTIVE_THRESHOLD = 0.35
- _HEAT_RESCUE_ACTIVE = 0.03
- _HEAT_CENTER_MARGIN = 0.20

TextReformatConfig fields:
- spacing
- binary_threshold
- blank_ink_ratio
- heat_rescue_peak

Blank-cell decision (priority order):
1. strong heat peak -> not blank
2. no ink fill -> blank
3. line-shaped ink -> blank unless heat_active >= rescue threshold
4. sparse ink -> blank if heat exists, otherwise keep conservatively
5. otherwise not blank

Important implementation note:
- heat stats are computed on center crop via _center_heat_stats
  to reduce neighbor-bleed false positives.

Text merge behavior:
- refine_text_raw detects title by leading blanks on first column
- restores spacing with full-width spaces
- apply_double_indent_heuristic upgrades likely 1-space indents to 2-space

## 6. FRONTEND MODULE DETAILS

### 6.1 App routing

Routes in App.tsx:
- / -> UploadPage
- /result -> ResultPage
- /batch -> BatchUploadPage
- /batch/result -> BatchResultPage

### 6.2 UploadPage

State machine:
- upload -> corner -> submitting

Flow:
1. detectCorners(file)
2. optional corner adjustment in CornerEditor
3. correctCorners(task_id, corners)
4. submitOCR(file, { task_id })
5. navigate to /result?taskId=...

### 6.3 ResultPage

- reads taskId from query
- uses useOCRProgress(taskId)
- shows ProgressBar while processing
- done state shows:
  - original image in ZoomableImage
  - OCR content (linear/manuscript toggle)
- supports copy-to-clipboard

### 6.4 BatchUploadPage

Steps:
- upload
- preparing
- review
- editing
- submitting

Behavior:
- accepts up to 50 images
- prepareBatch(files)
- confidence badges:
  - >=0.8 high
  - >=0.4 medium
  - <0.4 low (recommend manual adjustment)
- manual corner edits are local state only
- submitBatch({ corrections })
- passes imageUrls/filenames to BatchResultPage through location state

### 6.5 BatchResultPage

- reads batchId from query
- uses useBatchProgress(batchId)
- shows aggregate progress and per-task cards
- supports:
  - copy one
  - copy all
  - export txt/json via ExportMenu

### 6.6 hooks/useOCRProgress.ts

- opens WS /ws/{taskId}
- polling fallback every 3000ms
- on WS status completed/failed:
  - triggers final poll immediately
  - cleans up ws + interval

### 6.7 hooks/useBatchProgress.ts

- opens WS /ws/batch/{batchId}
- polling fallback every 3000ms
- on WS status completed:
  - fetches full batch status immediately

### 6.8 frontend/lib/api.ts

Health:
- fetchHealth

Single OCR:
- detectCorners
- correctCorners
- submitOCR
- pollOCRStatus
- connectProgressWS

Batch OCR:
- prepareBatch
- submitBatch
- pollBatchStatus
- connectBatchProgressWS

Important typing mismatch note:
- CornerDetectResponse in frontend types does not include task_id,
  so detectCorners returns an augmented type with task_id manually extracted.

### 6.9 frontend/lib/export.ts

Single export:
- exportSingleTxt

Batch export:

## 9. KNOWN ISSUES / RISKS (CURRENT)

1. Persistence gap
- _tasks, _pending_images, _batches are in-memory only.
- restart loses all running/prepared jobs.

2. CORS is fully open
- allow_origins=["*"] in production is risky.

3. Batch submit filename gap
- BatchSubmitResponse currently returns filename="" for each task.

4. Documentation drift risk
- Some historical docs reference v3.1.0, but runtime code is 3.0.0.

5. Frontend type mismatch around corner detect task_id
- helper compensates at runtime, but type contract is inconsistent.

## 10. AGENT OPERATION RULES FOR THIS REPO

When modifying code, follow these repo-specific rules:

1. If backend schema changes:
- update backend/api/schemas.py
- update frontend/src/lib/types.ts
- verify frontend/src/lib/api.ts callers

2. If route paths/params change:
- update backend route file
- update frontend lib/api.ts
- update page/hook callers

3. If progress payload changes:
- update backend ws schemas and broadcaster
- update both hooks: useOCRProgress and useBatchProgress

4. If text_reformat logic changes:
- run/expand backend/tests/test_text_reformat.py
- preserve center-heat crop intent unless intentionally replaced

5. If changing task storage model (e.g. Redis):
- migrate _tasks/_pending_images/_batches behavior consistently
- keep single and batch progress behavior equivalent

## 11. QUICK RECIPES

### Add a new VLM backend
1. Extend VLMSettings in backend/config/settings.py
2. Implement strategy in backend/services/vlm_service.py
3. Wire selection in factory method
4. Update env docs and any prompt/model assumptions

### Add a new backend endpoint
1. Add route under backend/api/routes/
2. Define request/response in backend/api/schemas.py
3. Register router in backend/api/app.py if new file
4. Add client function in frontend/src/lib/api.ts
5. Add/update frontend types in frontend/src/lib/types.ts

### Add a new OCR stage
1. Integrate stage in backend/services/ocr_pipeline.py
2. Emit on_progress stage/progress
3. Ensure frontend hooks display new stage safely
4. Update tests/docs

### Harden for production
1. Add persistent task store (Redis)
2. Restrict CORS origins
3. Add CI: ruff + pytest + frontend lint/build
4. Validate docker compose on clean machine
